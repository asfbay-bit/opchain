## Goals (why redesign & reimplement)

The reverse-spec identified 4 HIGH and 10 MED gaps. Those can be addressed by
incremental patches, but the site also has structural issues that incremental
work won't fix:

1. **Catalog drift risk** — three separate sources of truth for the skill list
   (`public/skills.js`, `src/opchain-try.js` `SKILL_PROMPTS`, `public/tryit.js`
   `STARTERS`/`INTROS`).
2. **Stub pages** — Architecture and Install are one-paragraph placeholders
   today. The reverse-spec output is already richer than anything on the live
   site.
3. **No type safety, no tests, no CI** — can't safely iterate on the Worker or
   on client JS.
4. **Implicit contracts everywhere** — handlers accept arbitrary JSON;
   Worker and client hand-serialize shapes; no schema to lean on.
5. **Design language is thin** — one-font, single-theme, no iconography, no
   visual system beyond color tokens. Stub pages and bare headers make the
   site feel unfinished.

A redesign + reimplementation lets us collapse the catalog into one source of
truth, generate the static pages from real content (the reverse-spec docs),
move to TypeScript, add CI, and introduce a stronger visual language — all in
a coherent cutover rather than a series of drift-prone patches.

## Success Criteria (measurable)

At the end of the roadmap, the following are true:

- **Single source of truth:** editing `skills/<skill>/SKILL.md` frontmatter is
  the only place a skill's display metadata lives. The site, the Try-It demo,
  and the bundled ZIP all derive from it at build time.
- **TypeScript everywhere:** Worker + any client modules are `.ts` with no
  `any` in exported surfaces; `tsc --noEmit` runs in CI.
- **Tests:** unit tests for all pure functions (email validation, HMAC
  round-trip, label/priority maps, markdown render); integration tests
  (Miniflare or `unstable_dev`) for every API endpoint; minimum
  coverage target 70%.
- **CI gates:** GitHub Actions runs build + test + type-check on every PR;
  `main` is protected. Staging deploy preview per-PR.
- **Staging environment:** `opchain-staging` Worker on a subdomain
  (e.g. `staging.opchain.dev`) with its own KV preview id; no accidental
  prod writes during `wrangler dev`.
- **No hardcoded secrets or fallbacks:** `DEPLOY_API_TOKEN` is required at
  startup; absence is a 500 at request time.
- **Security headers:** CSP + Referrer-Policy + X-Frame-Options + Permissions-Policy
  set on every static response, in addition to the existing HSTS + nosniff.
- **Content parity + expansion:** Introduction, Architecture, Skill Library,
  Install, Try It pages all have real content (architecture includes the
  mermaid diagrams from the reverse-spec; install has CLI + Claude.ai +
  Claude Desktop instructions; every skill has its own detail page).
- **Design system v2:** documented tokens, typography scale, component
  library, styleguide page, focus-visible treatment across all interactive
  elements.
- **Performance:** static pages Lighthouse Performance ≥ 95 and Accessibility ≥ 95;
  Try-It chat TTFB < 500 ms on the streaming start event.
- **Migration:** zero visitor-visible downtime during cutover; old URLs still
  resolve.
- **Analytics funnel visibility:** Cloudflare Web Analytics captures page
  views; PostHog captures the funnel (landing → email gate → first
  exchange → install click → ZIP download → feedback submit). Consent
  banner present and pre-event (PostHog cookies only set after accept).

## Non-goals (explicit out-of-scope)

- User accounts, login, billing.
- A full CMS — content is markdown in the repo.
- Translations / i18n.
- A mobile app.
- Re-architecting the skills *themselves* (the SKILL.md files). This roadmap
  only touches the presentation surface + the Worker API.
- Session replay recordings (PostHog feature, off by default).
- A/B testing / feature flags (PostHog supports both; not planned here).
