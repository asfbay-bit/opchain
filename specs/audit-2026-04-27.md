## What's working well (called out so we don't quietly remove it)

1. **CSP is comprehensive** (`src/index.js:101-113`). nonce + `strict-dynamic`,
   tight `connect-src`, `frame-ancestors 'none'`, `base-uri 'self'`,
   `form-action 'self'`. The only relaxation is `style-src 'unsafe-inline'`
   for Tailwind 4 inline styles — explicitly tagged as backlog item B-08.
2. **Baseline security headers stamped on every response** (`src/index.js:115-120`,
   re-applied at the outermost wrapper at `:344-349`). Idempotent on purpose —
   even if a future handler forgets, the outer wrapper covers it.
3. **HMAC token verification fails closed** (`src/opchain-try.js:149-152, 182-185`).
   Missing `DEPLOY_API_TOKEN` returns 503; no fallback, no implicit secret.
4. **Zod schemas enforce input shape at every POST endpoint** (`src/lib/schemas.js`).
   Error shape is consistent (`{ error, code, issues? }`).
5. **Markdown rendering is sanitized** (`site/src/lib/markdown.ts`).
   marked + isomorphic-dompurify + a DOMPurify hook that strips non-safe URL
   schemes and forces `rel="noopener noreferrer"` on links.
6. **Outbound retry is bounded** (`src/lib/retry.js`). 4xx never retried, 5xx
   retried once with jittered backoff, network errors retried once.
7. **Analytics is fire-and-forget** (`src/lib/analytics.js`, called via
   `ctx.waitUntil`). Failures swallowed; never blocks user-visible flow.
8. **Structured logging** (`src/lib/request-id.js`). One JSON line per event
   with stable event names in `EVENTS`. Cloudflare Logs + downstream ingestion
   parse without regex.
9. **Tests exercise the critical paths** (`tests/feedback.test.js`,
   `tests/health.test.js`, `tests/security-headers.test.js`,
   `tests/csp-nonce.test.js`, `tests/validation.test.js`,
   `tests/lead-ttl.test.js`, etc.).

## Findings

### HIGH

_None._

### MEDIUM

#### A-M1. Redundant `String()` cast and `slice(0, 4000)` after Zod validation
**File:** `src/opchain-try.js:235`
**Issue:** Zod already enforces `content: string().min(1).max(4000)` via
`ChatMessage`. The `cleanMessages` map redoes both as `String(m.content).slice(0, 4000)`.
Defensive coding is fine, but the redundancy implies the validation contract
isn't trusted. Either drop the cast, or document why both layers exist
(future-proofing against schema changes).
**Severity:** MED (code quality, not correctness)
**Suggested fix:** Replace with `messages.slice(-(MAX_EXCHANGES * 2))` and trust Zod.

#### A-M2. Conversation truncation can drop context mid-thread
**File:** `src/opchain-try.js:236`
**Issue:** `cleanMessages.slice(-(MAX_EXCHANGES * 2))` keeps the last 10 messages.
For a 5-exchange demo that's fine, but if Zod's max(20) is ever bumped or the
client sends interleaved roles outside the user/assistant pair pattern, the
truncation can sever an assistant message from its preceding user prompt.
**Severity:** MED (UX correctness)
**Suggested fix:** Slice in *exchange pairs* — find the index that keeps complete
user→assistant pairs ending with the most recent user message.

#### A-M3. IP rate-limit increment fires before email-cap check
**File:** `src/opchain-try.js:127-145`
**Issue:** `writeIpWindow` increments the IP counter, then the email-exhaustion
check runs. A user exhausted on email exchanges will still consume IP bucket
slots on every retry (subject to KV write costs and bucket TTL).
**Severity:** MED (defensible, but undocumented)
**Suggested fix:** Either add a comment explaining the order is intentional
(prevents abuse via email rotation), or move the IP increment after both checks.

#### A-M4. CORS `Access-Control-Allow-Methods` over-broad per route
**File:** `src/index.js:60-70`
**Issue:** Single `corsHeaders` returns `POST, GET, OPTIONS` for every CORS-eligible
route, but `/api/health` is GET-only and `/api/feedback` + `/api/try/*` are
POST-only. The Allow-Methods header advertises a method for each route that
isn't valid on that route.
**Severity:** MED (low-impact, but technically inaccurate)
**Suggested fix:** Pass per-route allowed methods into `corsHeaders`, default to
the union for backwards compatibility.

#### A-M5. `applySecurityHeaders` reads full HTML body into memory
**File:** `src/index.js:122-141`
**Issue:** Nonce substitution requires a full-body buffer (`response.text()`).
For Astro static pages this is sub-100kB so it's fine, but loses the streaming
benefit on any future large-HTML response.
**Severity:** MED (perf, only matters if HTML grows)
**Suggested fix:** Note this constraint in the comment; if HTML grows past
~500kB consider a streaming `TransformStream` that scans for the placeholder.

#### A-M6. Two parallel logging surfaces in `bindLogger`
**File:** `src/lib/request-id.js:18-30`
**Issue:** `info`/`warn`/`error` (free-form) and `event`/`eventWarn`/`eventError`
(structured JSON) coexist with no guidance on when to use which. Mix is visible
in handlers (`opchain-try.js` mixes `log.warn("analytics error:", e.message)`
with `log.eventError(EVENTS.UPSTREAM_FAILED, ...)`).
**Severity:** MED (observability)
**Suggested fix:** Pick one: structured-only (drop the free-form), or document
that free-form is "messages that don't fit a schema" and lint for it.

### LOW

#### A-L1. Hardcoded Linear label IDs
**File:** `src/index.js:28-33`
**Issue:** `LABEL_MAP` UUIDs are baked into source. Team/project IDs have env
overrides; label IDs do not. Renaming a Linear label requires a code change.
**Severity:** LOW
**Suggested fix:** Accept a JSON env var `LINEAR_LABEL_MAP` falling back to the
hardcoded defaults.

#### A-L2. `getClientIP` fallback bucket
**File:** `src/opchain-try.js:107-113`
**Issue:** Unknown IP falls back to `0.0.0.0`, so all unknown-IP requests share
one bucket. Real-world this almost never fires (CF-Connecting-IP is reliable on
Workers), but worth a comment.
**Severity:** LOW
**Suggested fix:** One-line comment explaining the bucket-collision is intentional
DoS protection.

#### A-L3. `retry.js` reachable-but-flagged unreachable line
**File:** `src/lib/retry.js:26-27`
**Issue:** Comment says "Unreachable" but the static control flow can reach it.
Coverage tools may flag this.
**Severity:** LOW
**Suggested fix:** Either remove the line and the comment (the throw above is
guaranteed by the loop logic), or replace with `/* c8 ignore next */` directive.

#### A-L4. `corsHeaders` returns header for unknown origins silently
**File:** `src/index.js:66-68`
**Issue:** When origin is not in `ALLOWED_ORIGINS`, `Access-Control-Allow-Origin`
is omitted from the response, which is correct browser-side. But there is no
log line for the rejection, so a misconfigured client gets a confusing CORS
failure with no server-side trace.
**Severity:** LOW
**Suggested fix:** Add `log.event("cors_rejected", { origin })` for `/api/*`
requests with disallowed origin.

#### A-L5. `LABEL_MAP` and `PRIORITY_MAP` exported but only consumed by tests
**File:** `src/index.js:28-35`
**Issue:** Verify these exports are tests-only. If so, fine. If app code consumes
them elsewhere, fine. The export hygiene is worth a one-time audit so internal
constants don't drift into the public surface.
**Severity:** LOW
**Suggested fix:** None if test-only; consider `// @internal` comment.

#### A-L6. `analytics.js` env-gated on POSTHOG_PROJECT_API_KEY only
**File:** `src/lib/analytics.js:38-41`
**Issue:** Single env-gate disables ALL analytics. If a future event becomes
mission-critical (e.g. "feedback_submitted" → triage queue), it would silently
fall back to no-op.
**Severity:** LOW
**Suggested fix:** Document the gate with a comment, or split mission-critical
events to a separate transport.

#### A-L7. DOMPurify hook installed once per module
**File:** `site/src/lib/markdown.ts:16-31`
**Issue:** `hookInstalled` is module-level. In Workers, modules are shared
across requests; the hook persists. The hook is read-only and pure so this is
benign, but the contract should be documented.
**Severity:** LOW
**Suggested fix:** Comment: "Hook is per-module-instance, shared across requests
in Workers. Hook is idempotent — installing twice is harmless."

#### A-L8. `cleanMessages` does not validate message ordering
**File:** `src/opchain-try.js:234-236`
**Issue:** Zod schema allows any sequence of `user`/`assistant` messages.
Anthropic Messages API requires strict alternation starting with `user`. A
malformed transcript will hit a 400 from Anthropic, surfacing as
`upstream_error` 502 to the client.
**Severity:** LOW (Anthropic catches this; user sees a generic error)
**Suggested fix:** Add a Zod refinement (or runtime check) asserting alternation
and starting role; return `invalid_messages` 400 directly with a clearer
message.
