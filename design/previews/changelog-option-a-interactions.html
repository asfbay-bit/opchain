v1.4.2 — Complete bundles, honest checkpoints

Jun 05, 2026 emergency patch 18 skills back-compatible

An out-of-band emergency patch: one user-facing breakage and a long-overdue correctness pass, shipped ahead of v1.5. The breakage — the "download all" skill bundle was shipping only each skill's SKILL.md, silently dropping the references/, scripts/, and examples/ that make a skill actually run.

What's new

  • "Download all" ships complete skills again. opchain-skills.zip now bundles full skill trees, matching the per-skill downloads on /skills.
  • oc- canonical skill names. Every skill gains an oc-<name> alias — oc-app-architect, oc-git-ops, oc-deploy-ops — so opchain skills are unambiguous next to same-named local skills.
  • The orchestrator stops misreporting itself. Persistence docs rewritten to match the git-tracked .checkpoints/orchestrator.checkpoint.json it actually uses.
  • One checkpoint schema, actually enforced. Validator, spec, and README now agree on a single wire schema. next_actions required while in-progress.
  • Drift detection & merge reliability. checkpoint:doctor added, cross-checks against git log and live /api/health.

New tooling

  • checkpoint:doctor — cross-checks checkpoints against git log + live /api/health
  • checkpoint:next — registry-free "what should I do next" for the local project
  • checkpoint:status --brief / --since=<date> — terse resume view + momentum digest
  • checkpoint:init — deterministic cold-start scaffold
Back-compatible. No migration required. The oc-<name> names are additive aliases. Checkpoint wire schema stays pinned at protocol_version 1.0.

v1.4 — Coverage as a plugin registry

May 12, 2026 pack registry 32 packs 5 skills bumped

opchain v1.4 turns stack-forge's coverage from hardcoded prose into a versioned pack registry — one pack.yml per language, framework, mobile platform, or deploy target, validated on every build. Total coverage grows from 6 stacks in v1.3 to 32 packs in v1.4.

What's new

  • Pack registry architecture. /coverage is the user-visible catalog. scripts/gen-stack-packs.mjs validates every pack on every build.
  • Hybrid dispatch. deploy-ops reads packs at runtime; api-dev reads packs at build time for typed scaffold metadata.
  • kind:mobile dispatch. Mobile packs render the ref doc as a release checklist via dispatchMobile().
  • Modern web bulk. elixir / bun / deno language packs; phoenix / remix / sveltekit / solid framework packs.
  • Enterprise bulk. java / csharp / kotlin / php; spring, dotnet, laravel framework packs.
  • Apple/iOS. swift + swiftui + ios-swiftui mobile pack + app-store deploy adapter.
Back-compatible with v1.3. Five skills version-bump in lockstep: stack-forge, api-dev, deploy-ops, app-architect, migration-ops.

v1.3 — Runtime PM, real platforms, automated releases

May 11, 2026 release-ops 18th skill PM-MCP

opchain v1.3 makes the v1.2 PM-MCP prose executable end-to-end, expands the platform menu beyond JS / Cloudflare, and ships release-ops — the 18th skill — to automate the "scope → /changelog → bump → ship" cadence opchain uses for itself.

What's new

  • The PM-MCP loop is real. A concrete tool-name registry, retry / backoff, idempotency markers, and a deferred-action queue.
  • release-ops is the 18th skill. Verbs: /release plan|draft|bump|announce|ship.
  • Platform menu grew. Django + Postgres + Render, Rails + Postgres + Heroku, Go + Fly.io, Rust + Axum + Shuttle.rs.
  • Build-time PM-MCP validator. Blocks the build on placeholder drift or unknown tool names.
Back-compatible with v1.2. All 18 skills lockstep at 1.3.0.

v1.2 — PM-tool MCP integration

May 05, 2026 PM-MCP Linear Jira

opchain v1.2 wires the skills into the PM-tool MCPs Anthropic ships with Claude Code. The ticket is the thread of execution, not a side artefact.

What's new

  • app-architect /discover accepts a ticket reference and pre-fills the interview from the ticket body.
  • git-ops /git-sync TICKET-1234 shapes the branch / commit / PR from the ticket and transitions ticket state on PR open and merge.
  • deploy-ops auto-creates a deploy ticket per environment, parent-linked to every commit's tickets.
  • monitoring-ops opens an incident ticket when alerts fire, parent-linked to the most recent deploy ticket.
Back-compatible with v1.1. PM loop is enrichment, never a hard dependency.
