v1.4.2 — Complete bundles, honest checkpoints

Jun 05, 2026emergency patch18 skillsback-compatible

An out-of-band emergency patch: one user-facing breakage and a long-overdue correctness pass, shipped ahead of v1.5. The breakage — the "download all" skill bundle was shipping only each skill's SKILL.md, silently dropping the references/, scripts/, and examples/ that make a skill actually run.

What's new
  • "Download all" ships complete skills again. opchain-skills.zip now bundles full skill trees, matching the per-skill downloads on /skills.
  • oc- canonical skill names. Every skill gains an oc-<name> alias — oc-app-architect, oc-git-ops, oc-deploy-ops — so opchain skills are unambiguous next to same-named local skills. Bare names still resolve.
  • The orchestrator stops misreporting itself. Persistence docs rewritten to the git-tracked .checkpoints/orchestrator.checkpoint.json it actually uses. The phantom checkpoint.sh is replaced by scripts/checkpoint.mjs.
  • One checkpoint schema, actually enforced. Validator, spec, and README now agree on a single wire schema. next_actions required while in-progress. Timestamp-sanity and referential checks added to validator.
  • Drift detection & merge reliability. Orchestrator logged three manual reconciliations in three weeks. v1.4.2 adds automated drift detection; stale checkpoints can no longer tell future-you to redo shipped work.
New CLI commands
  • checkpoint:doctor — cross-checks checkpoints against git log + live /api/health, flags drift
  • checkpoint:next — registry-free "what should I do next" for the local project
  • checkpoint:status --brief / --since=<date> — terse resume view + momentum digest with stale-age flag
  • checkpoint:init — deterministic cold-start scaffold
  • checkpoint -- done <skill> — pop top next_action into recently_done log
Back-compatible with v1.4. The oc-<name> names are additive aliases. Checkpoint wire schema stays pinned at protocol_version 1.0. All 18 skills lockstep-bump to 1.4.2.

v1.4 — Coverage as a plugin registry

May 12, 2026pack registry32 packs5 skills bumped

opchain v1.4 turns stack-forge's coverage from hardcoded prose into a versioned pack registry — one pack.yml per language, framework, mobile platform, or deploy target, validated on every build. Total coverage grows from 6 stacks in v1.3 to 32 packs in v1.4.

What's new
  • Pack registry architecture. scripts/gen-stack-packs.mjs validates every pack on every build. Coverage-flag emitted per language / framework / mobile pack.
  • Hybrid dispatch. deploy-ops reads packs at runtime; api-dev reads packs at build time for typed scaffold metadata.
  • kind:mobile dispatch. Mobile packs render ref docs as release checklists via dispatchMobile(), not executed commands.
  • Modern web bulk. elixir / bun / deno; phoenix / remix / sveltekit / solid packs.
  • Enterprise bulk. java / csharp / kotlin / php; spring / dotnet / laravel packs. Canonical lint commands: mvn checkstyle:check, dotnet format --verify-no-changes, ktlint, phpcs.
  • Apple/iOS. swift + swiftui + ios-swiftui mobile pack + app-store deploy adapter. First production user of dispatchMobile().
  • Mobile bulk. kotlin-android / flutter / react-native-expo + play-store deploy adapter.
  • Hosting adapters. railway / netlify / heroku / aws-amplify deploy-target packs.
The pipeline that shipped it

v1.4 went through opchain itself. PRs 4-7 ran in parallel from sibling worktrees on the same origin/main base, opened concurrently, merged sequentially with rebases. release-ops drafted this changelog entry from the merged PR summaries.

Back-compatible with v1.3. Five skills bump in lockstep: stack-forge, api-dev, deploy-ops, app-architect, migration-ops.

v1.3 — Runtime PM, real platforms, automated releases

May 11, 2026release-ops18th skillPM-MCP live

opchain v1.3 makes the v1.2 PM-MCP prose executable end-to-end and ships release-ops — the 18th skill — to automate the "scope → /changelog → bump → ship" cadence opchain uses for itself.

What's new
  • The PM-MCP loop is real. Concrete tool-name registry, retry/backoff, idempotency markers, deferred-action queue — skills can crash-restart mid-flow without polluting Linear/Jira/GitHub Issues.
  • release-ops is the 18th skill. /release plan|draft|bump|announce|ship. Reads checkpoints to propose next semver, drafts changelog entry, bumps all skill versions atomically.
  • Platform menu grew. Django + Postgres + Render, Rails + Postgres + Heroku, Go + Fly.io, Rust + Axum + Shuttle.rs.
  • Build-time PM-MCP validator. Blocks the build on placeholder drift, missing protocol citations, or unknown tool names.

v1.5 — Build the AI app

built · awaiting releasein progress22 skillsAI-native

v1.5 closes the biggest categorical gap in the catalog: nothing today owns AI-feature design. v1.5 adds four new AI-native skills wired through the existing pipeline. Markdown methodology turns out to be exactly the right shape for the things AI apps need most: source-controlled, diffable, version-pinned prompts and eval sets.

New skills
  • claude-api — /claude-api. Model selection (Opus / Sonnet / Haiku routing), prompt caching as a default, batch vs. streaming, tool use, and /claude-api migrate playbooks for model-version bumps (4.6 → 4.7, retired-model replacement). Cache-audit checks hit rate; cost guardrails cap tokens per phase.
  • rag-forge — /rag. Tri-agent (Designer / Builder / Evaluator). Vector-DB decision tree, embedding-model choice, chunking strategy, hybrid search, and retrieval eval (/rag eval, /rag bench) with golden sets and recall@k / NDCG.
  • agent-forge — /agent. Tri-agent (Planner / Builder / Evaluator). Claude Agent SDK scaffold, subagent topology, tool-budget design, harness-loop shapes (chat / autonomous / human-in-the-loop), and eval harness over golden trajectories.
  • prompt-ops — /prompt. Prompts as first-class artifacts: file-shaped versioning, eval datasets, and regression detection. /prompt regress runs the golden set on PR-changed prompts and blocks on a drop.
Cross-skill ripples
  • app-architect gains an "AI app?" branch in /discover that routes to the four new skills.
  • stack-forge adds vector-DB packs (pgvector, Turbopuffer, Pinecone, Supabase Vectors) to the pack registry.
  • code-auditor learns prompt-injection and tool-use safety rules.
Site & GTM

New /ai-recipes walkthroughs, a "Build AI apps" row on /compare, an AI-app variant of /pipeline-builder, the /blog dev-log scaffold with launch posts, a /for-agencies page, and the newsletter flag wired. The /demo Worker is rewritten on claude-api patterns.

Back-compatible with v1.4. No migration required. All 22 skills version-bump to 1.5.0. Four new skills ship gated behind their skills.registry.<id>.enabled flags.

v1.6 — The instrumented pipeline

planned · est. Jul 2026planned2 new skillscheckpoint v1.3

v1.5 introduces four AI-native skills; the predictable next question is what did they cost, and is my pipeline actually getting better? v1.6 answers both. It adds cost attribution and opt-in telemetry, turns every skill's pass/fail into a graded eval score, and ships the first checkpoint-protocol bump since v1.2.

New skills
  • cost-ops — /cost. Attributes Anthropic spend to each skill phase, adds budget gates to checkpoints, and recommends model-tier routing — Haiku for cheap repetitive phases, Opus for spec and audit. Finally answers "what did this feature cost to ship?"
  • telemetry-ops — /telemetry. Opt-in, local-first usage metering written to .checkpoints/usage.sqlite. Aggregates anonymized "which skills people actually use" stats — the data behind the new public /dashboard. Nothing leaves your machine unless you opt in.
Cross-skill ripples
  • bug-check + code-auditor stop emitting bare pass/fail and start scoring against a stable rubric.
  • monitoring-ops gains an "AI-app" template covering token rate, eval drift, and hallucination flags.
  • orchestrator's /ops next factors cost and remaining budget into what it recommends.
Protocol bump

checkpoint-protocol v1.3 — the first bump since v1.2 PM-MCP fields — adds cost, eval_scores, and telemetry_handle. migration-ops rolls it across every skill the same way the governance frontmatter rolled out in v1.4.

Back-compatible with v1.5. All 24 skills lockstep-bump to 1.6.0.

v1.7 — Pick one distribution play

planned · lean: marketplace + templatesplannedest. Sep 2026

By the time v1.6 ships, opchain has 24 skills, instrumented pipelines, and AI-native coverage. The biggest remaining gap isn't features — it's distribution. v1.7 deliberately promotes one growth play into a flagship theme instead of accreting more skills. The current lean is marketplace + templates.

The lean plan
  • marketplace. A site surface for community-contributed skills and packs. Every contribution adds breadth at zero core-team cost — and opchain itself becomes the first entry in its own marketplace. Compounds: the natural follow-on once v1.6's telemetry exists to grade community contributions on real usage.
  • template-ops — /template. Opinionated project starters that pre-wire an entire pipeline — stack, skills, and checkpoints — so a new project is ready to run from the first commit.
Alternatives on the table
  • Agency play. monorepo-ops for real multi-project checkpoint scoping, a /for-agencies expansion, and a client-handoff demo scenario.
  • Pipeline depth. discovery-ops (JTBD / opportunity trees, upstream of app-architect) plus qa-ops splitting test-pyramid design out of bug-check.
GTM

A marketplace launch post, a /case-studies page with named (or pseudonymous) builds, and a /community hub.

Lean, not locked — the final v1.7 theme is chosen after the v1.5 / v1.6 retro.
