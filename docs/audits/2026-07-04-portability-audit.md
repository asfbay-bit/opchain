## Ground-truth investigations

### (checkpoint-protocol)

Deep-read of oc-checkpoint-protocol (SKILL.md + INTEGRATION.md + WALKTHROUGH.md), all 17 bundled references/checkpoint-protocol.md copies (byte-identical, md5 675a28bf), orchestrator.md, .checkpoints/README.md, scripts/checkpoint.mjs, and the checkpoint sections of oc-app-architect/oc-git-ops/oc-deploy-ops/oc-orchestrator. Verdict: a fresh user machine can only write a checkpoint by improvising OUTSIDE the docs — there is no imperative "mkdir .checkpoints; write this JSON with the Write tool" anywhere in the shipped skills (grep -rn mkdir skills/ returns zero hits); every documented creation path (init), update path (update/done), and validation path (npm run checkpoint:validate) shells out to scripts/checkpoint.mjs, which lives only at the private repo root, is explicitly declared the sole canonical writer ("skills do not bundle their own writers"), and is excluded from all three distribution channels (combined zip, per-skill zips, public mirror). The protocol further steers the model away from the one workable method: WALKTHROUGH.md says "no hand-editing JSON". The schema itself ships in 19 of 27 skills (17 verbatim copies + the protocol SKILL.md + one divergent stale copy in oc-dash-forge) but is entirely absent from 8 skills despite orchestrator.md twice claiming it is "bundled in every skill"; there is also a wire-version self-contradiction (prose mandates stamping "1.1", the only concrete schema example in every copy stamps "1.0"). This fully explains field symptom (1): on user machines the first write attempt is a failed `node scripts/checkpoint.mjs init`, the mandated post-write validator is a missing npm script, and the fallback the model would naturally reach for is documented as an anti-pattern.

### (distribution)

Built the real artifacts (scripts/make-skills-zip.sh) and audited what users actually receive across all three channels plus the install docs and build gates. The zips have correct top-level layout (unzip -d .claude/skills/ nests skill dirs correctly), but the checkpoint protocol's canonical writer (scripts/checkpoint.mjs) is not distributed in ANY channel while the shipped docs route all checkpoint writes and cold-start scaffolding through it — directly explaining "checkpoints not written" in the field. Separately, 10 of 27 skills ship with no references/orchestrator.md or checkpoint-protocol.md (the sync tool only updates files that already exist and the build validator never checks referenced files exist), explaining unreliable cross-skill chaining. The mirror does include the top-level skills/orchestrator.md (cp -R skills), but the combined zip does not.

### (drift)

Compared shipped skills/ against the author's repo-local .claude/skills (a pure symlink farm since commit 3482c71 — zero content drift, single diff is the un-symlinked CHANGELOG.md), the author's personal ~/.claude (no opchain skills installed at all), the live zip at opchain.dev (27 skills, all 1.7.0, deploy current at 05c007f == main HEAD), and the public mirror (@1d2cbbd, current). Conclusion: the author is testing exactly the content users get, so the field failures are environmental, not version drift. The author's repo carries a Stop hook that blocks session end until checkpoints are written and a PreToolUse hook that blocks git commit until oc-bug-check passes — the two exact behaviors reported broken in the field — plus the scripts/checkpoint.mjs CLI that the shipped protocol doc names as the canonical (and only permitted) checkpoint writer; none of this ships in any channel. Secondary drift: 10/27 skills ship without the bundled orchestrator/checkpoint protocol files (masked in-repo by the shared top-level copies), and the zip channel omits skills/orchestrator.md and skills/CHANGELOG.md that the mirror includes.

### (sim-checkpoint)

Simulated a fresh user machine (zip-equivalent install of all 27 skills into ~/.claude/skills, empty git project) and traced the checkpoint-write path literally for oc-app-architect, oc-code-auditor, and oc-git-ops. Verdict: checkpoints are SOMETIMES written and essentially NEVER persisted — every documented write mechanism (scripts/checkpoint.mjs, npm run checkpoint:*, init scaffold, pre-commit hook, 'copy from the reference repo') fails with MODULE_NOT_FOUND/missing-script because the tooling lives only in the private repo, the protocol explicitly forbids skills bundling their own writers, and no fallback instruction exists; a file materializes only if the model improvises hand-written JSON. Worse, even hand-written checkpoints are immediately gitignored by two shipped instructions (oc-app-architect scaffold and oc-git-ops '.gitignore Enforcement', both contradicting the bundled protocol's 'Do not add .checkpoints/ to .gitignore'), so state never survives across machines or clones — which fully explains the field reports.

### (sim-chaining)

Simulated both installs in a sandbox (full suite: 27 skill dirs; single: oc-git-ops only) and traced every hop against shipped text. FULL install verdict: chaining DEGRADES — exactly one hop is deterministic (oc-git-ops → oc-bug-check via an explicit Skill() call at oc-git-ops/SKILL.md:228); the sprint-pass → oc-git-ops and git-sync → oc-deploy-ops hops are demoted to passive "suggest" by the skills' own text, and every "read that skill's SKILL.md" instruction uses repo-shaped paths (oc-stack-forge/SKILL.md) that resolve nowhere after install, leaving discovery to probabilistic description-keyword triggering. SINGLE install verdict: BREAKS — the mandatory pre-commit gate calls Skill(skill="oc-bug-check") for a skill that isn't installed, the shipped table's only branch for that state is "invoke it first" (a closed loop), ad-hoc checks are explicitly forbidden, and a suite-wide grep confirms there is no "if the skill isn't installed" fallback anywhere. The four no-references skills never load the protocol at all, and oc-orchestrator — the router — has a boot sequence (npm run checkpoint:status / node scripts/checkpoint.mjs init) that cannot complete on any machine except this repo, while being ordered not to route until it does.
